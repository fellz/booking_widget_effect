# booking_widget (Vue 3) → booking_widget_effect (foldkit)

> 🇬🇧 English version: [COMPARISON.md](./COMPARISON.md)

Переписывание одного и того же виджета бронирования номера один в один, чтобы
ответить на один вопрос: **сколько дыр корректности оригинала закрывает
архитектура TEA + Effect (foldkit) — структурно, а не «если не забыть быть
аккуратным»?**

Тот же домен, те же детерминированные данные о доступности, тот же флоу (даты →
гости → номер → проверка → подтверждение), та же i18n ru/en, переключатель темы,
календарь и граница API (mock/http). Разница целиком в том, *как моделируется
состояние и как выражаются переходы*.

- Версия на Vue: `../booking_widget`
- Версия на foldkit: этот проект (`src/`)

Запуск: `npm run dev`. Проверка: `npm test` (18 тестов), `npm run typecheck`, `npm run build`.

---

## Архитектура: полноценный TEA

Это не «Vue, припорошённый Effect» — это полная реализация **The Elm Architecture**,
петли, которую предписывает foldkit. Всё остальное в этом документе — следствие этих
пяти частей:

- **Model** — всё состояние приложения в одной иммутабельной структуре,
  типизированной `Schema` (`src/model.ts`). Никакого локального состояния
  компонентов, никаких ref.
- **Message** — каждый вход (клики, результаты команд) — вариант одного
  tagged-union (`src/message.ts`). Ничего не происходит, что не было бы Message.
- **update** — единственная *чистая* функция `(Model, Message) → [Model, Command[]]`
  (`src/update.ts`), матчится исчерпывающе (`Match.tagsExhaustive`). Это
  единственное место переходов состояния, и она не трогает I/O.
- **view** — чистая функция `Model → Document` (`src/view/`) через `html` DSL
  foldkit. UI — это проекция Model; своего состояния держать не может.
- **Command** — сайд-эффекты это *значения*, возвращаемые из `update`, исполняемые
  рантаймом, который подаёт их результаты обратно как Message (`src/command.ts`).
  View никогда не зовёт API; он диспатчит Message, `update` возвращает Command.

Рантайм (`src/entry.ts` → `Runtime.makeApplication` / `Runtime.run`) замыкает петлю:
dispatch → `update` → пере-`view`, с `Flags` для стартовых входов (`today`, тема) и
Layer в `resources` для графа зависимостей команд. Тестируется через TEA-нативные
примитивы foldkit — `Story` (гонять `update` сообщениями, проверять Model + Commands)
и `Scene` (проверять `view`) — так что без DOM и фейковых таймеров.

**Какие части TEA / foldkit здесь использованы, а какие нет** (для сравнения честность
важна): использованы — Model, Message, чистый `update`, `view`, `Command`, `Flags`,
Layer в `resources`, оверлей DevTools, тесты `Story`/`Scene`. *Не* использованы —
foldkit также предлагает Subscriptions, Submodels, Managed Resources, типобезопасный
роутинг, Ports и модуль `fieldValidation`; этому виджету ничего из этого не нужно
(валидация — чистая функция над Model), поэтому они опущены, а не показаны напоказ.

## Слой ошибок (ось, ради которой Effect и сделан)

У Vue-виджета не было настоящего понятия *типа ошибки*. `loadRooms`/`loadRoomCalendar`
делали `try { … } catch { status = 'error' }` — любой вид падения (нет сети, 500,
битый JSON, номер заняли прямо из-под тебя) схлопывался в один булевоподобный
`'error'`, а `confirm()` вообще не мог упасть. Переписанная версия делает падения
**типизированным, исчерпывающим каналом** от края до края:

1. **Именованная ADT ошибок** (`src/domain/errors.ts`) — `NetworkError`,
   `ServerError({status})`, `RoomTaken`, через `Data.TaggedError`. `LoadError` и
   `SubmitError` — объединения, которые может породить каждая операция.
2. **Сервис несёт ошибки в своём канале `E`** — `submitBooking` это
   `Effect<string, SubmitError>`, а не `Effect<string>`. http-адаптер маппит каждое
   падение в его источнике: реджект fetch → `NetworkError`, не-2xx → `ServerError`,
   409 → `RoomTaken`, битое тело (`SchemaError` от декодера Schema) → `ServerError`.
   Никаких `as`, никаких проглоченных исключений.
3. **Одна исчерпывающая точка перевода** из мира Effect в мир TEA — `Effect.catchTags({ RoomTaken, NetworkError, ServerError })`
   в командах (`src/command.ts`). Добавление варианта ошибки в сервис — **ошибка
   компиляции** здесь, пока он не обработан. Каждый вариант сохраняет свою
   идентичность в Message `FailedSubmit`, несущем типизированный `reason`.
4. **`reason` доезжает как данные до модели** (`SubmitFailed` несёт
   `FailureReason`), и i18n рендерит *разный* текст под каждую причину
   (`submitErrors`, exhaustive по причине) — «номер только что заняли» ≠ «нет
   связи» ≠ «сервер недоступен» — вместо одного общего «не удалось отправить».

То есть граница между типизированными ошибками и message-driven рантаймом — это
единственная явная, исчерпывающе проверяемая функция; ровно то, чего не хватало
конструкции `try/catch → status flag` во Vue.

**Данные от сервера декодируются, а не принимаются на веру** (аналог Elm-декодеров).
HTTP-адаптеры не кастуют `response.json()` к типу — они прогоняют его через
`S.decodeUnknownEffect(schema)` (`readJson` в `src/services/httpClient.ts`), проверяя
против `S.Array(Room)`, `S.Array(S.String)`, `S.Struct({reference})`. Несовпадение
формы — это `SchemaError`, который маппится в `ServerError` как любое другое падение.
Решающее улучшение над рукописными декодерами `Json.Decode` из Elm: схема *и есть*
тип — тот же `Room`, что в Model, декодирует и провод — поэтому декодер и доменный тип
не могут разъехаться. (`Flags` рантайм тоже валидирует через Schema на старте.
Mock-адаптер возвращает уже типизированные значения, так что декодирование важно
только на реальной границе провода.) Vue-версия делала `(await res.json()) as
RoomDto[]` — непроверенное приведение, которое спокойно отдало бы битые данные в UI.

> Живая проверка ветки падения вскрыла и реальный баг, который система типов
> *не* поймала: кнопка retry в `SubmitFailed` была привязана к `ClickedConfirm`,
> но `update` реагировал на `ClickedConfirm` только из `Reviewing`, поэтому retry
> был молчаливым no-op. Починено (повторная отправка и из `SubmitFailed`) и
> зафиксировано регрессионным `Story`-тестом. Полезное напоминание: «невозможные
> состояния вон» закрывает баги *представления*, а не баги *логики переходов* —
> для них всё ещё нужны тесты.

## Архитектура и домены (Layers)

Vue-версия инвертировала свою единственную зависимость вручную: интерфейс
`BookingApi`, две реализации (`createMockBookingApi` / `createHttpBookingApi`) и
один `createBookingApi()`, выбирающий по `VITE_API_URL`. Это чистый DI — но это
один крупный порт, проводка ручная, и нет понятия *графа зависимостей* (http-клиент,
конфиг и порты данных — всё спутано в одной фабрике).

Версия на foldkit выражает ту же идею как типизированный граф **Effect Layers**,
разложенный по доменным границам (`src/services/`):

- **Доменные порты**, по сервису на каждый — `RoomCatalog` (читать каталог),
  `AvailabilityCalendar` (читать занятые даты номера), `Reservations` (поставить
  бронь — сторона записи, со своей ошибкой `RoomTaken`). Разделение одного
  `BookingApi` на три ограниченных порта означает, что команда зависит только от
  того, что использует, а асимметрия чтение/запись явна в типах.
- **Инфраструктура и конфиг — тоже сервисы** — `HttpClient` (единственное место,
  которое трогает `fetch`) и `AppConfig` (базовый URL, задержки мока). Это
  зависимости как любые другие: внедряются, а не читаются ad-hoc.
- **Граф зависимостей, собранный через `Layer`** — `AppConfig ← HttpClient ←
  {три HTTP-адаптера}`, скомпонованный через `Layer.mergeAll` (порты рядом) и
  `Layer.provide` (подать им зависимости). Mock- и HTTP-стек — это два
  альтернативных графа, производящих *одни и те же* три сервиса:

  ```ts
  const MockServices = Layer.mergeAll(RoomCatalogMock, AvailabilityCalendarMock, ReservationsMock)
    .pipe(Layer.provide(AppConfigLive))

  const HttpServices = Layer.mergeAll(RoomCatalogHttp, AvailabilityCalendarHttp, ReservationsHttp)
    .pipe(Layer.provide(HttpClientLive), Layer.provide(AppConfigLive))
  ```

Что это даёт сверх ручного DI:

- **Канал `R` — это контракт зависимостей, проверяемый типами.** `update`
  возвращает `Command<Message, never, RoomCatalog | AvailabilityCalendar |
  Reservations>`. Если команда использует сервис, который Layer в `resources`
  рантайма не предоставляет, — это ошибка компиляции; нельзя забыть что-то
  подключить. У Vue-фабрики такой проверки не было: пропущенная зависимость —
  сюрприз в рантайме.
- **Меняешь один узел, а не весь фид.** Можно подать реальный `Reservations`,
  оставив моковые чтения, или фейковый `HttpClient` в тесте, компонуя другие
  Layer'ы — без переписывания фабрики.
- **Эффекты конструирования и жизненный цикл ресурсов — первоклассны.**
  `Layer.effect` позволяет *конструированию* сервиса быть Effect'ом (прочитать
  конфиг, открыть соединение); scoped-Layer'ы получают acquire/release. Vue-фабрика
  могла лишь вернуть простые объекты.

Цена реальна: больше файлов и словарь Layer/`Context.Service` ради того, что
сегодня — демо из трёх эндпоинтов. Для приложения такого размера ручной порт был
нормален; граф Layer'ов начинает окупаться по мере роста числа сервисов, их
взаимозависимостей и потребности варьировать их по окружению/тестам.

## Чего это стоило

Закрытия не бесплатны; они оплачены **моделированием заранее**:

- Больше именованных состояний. В сторе Vue было ~8 ref; в модели foldkit ~5
  сумм-типов (`DateSelection`, `RoomsLoad`, `Availability`, `RoomSelection`,
  `Phase`) плюс smart-конструируемый `ValidBooking`. Ты объявляешь состояния,
  которые иначе отслеживал бы неявно.
- View многословнее, чем SFC-шаблоны Vue — явный `html` DSL со строками Tailwind
  вместо `<template>` + scoped CSS.
- Реальная кривая обучения Effect/Schema (сервисы как `Context.Service`, команды
  как значения, `Match.exhaustive`, `evo` для иммутабельных обновлений).

Выигрыш: баги, которые аудит *нашёл* в версии на Vue, в версии на foldkit — это в
основном то, что нельзя выразить; а немногие оставшиеся, пишущиеся руками,
навязаны exhaustiveness или централизованы в одной чистой функции, и весь флоу
покрыт быстрыми детерминированными тестами `Story`/`Scene` (без DOM, без
моков-на-время).

## Тесты (TEA-нативные, без монтирования DOM)

Раз TEA разделяет чистый `update` (логика переходов), чистый `view` (рендер) и
`Command` (эффекты-значения) — каждый слой тестируется изолированно: без
монтирования компонентов, без моков `fetch`, без фейковых таймеров. 18 тестов идут
~30 мс.

**`Story` — тесты машины состояний (`src/story.test.ts`).** Гоняешь последовательность
Message через настоящий `update` и проверяешь и новый Model, *и какие Commands он
вернул* — при этом команды **не исполняются**:

```ts
Story.story(
  update,
  Story.with(validModel),
  Story.message(ClickedConfirm()),
  Story.Command.expectExact(SubmitBooking),                 // эффект — это ЗНАЧЕНИЕ, его видно
  Story.model(m => expect(m.phase._tag).toBe('Submitting')),
  Story.Command.resolve(SubmitBooking, SucceededSubmit({ reference: 'BK-TEST' })), // инжектишь результат
  Story.model(m => expect(m.phase._tag).toBe('Confirmed')),
)
```

- **Эффекты инспектируемы** — нет API-модуля для мока. Проверяешь «пришла команда
  `SubmitBooking`», затем `resolve` подаёт её результат как Message, история идёт дальше.
- **Нет таймеров и `await`** — эффекты не выполняются, ждать нечего.
- **Гонки детерминированы** — тест «устаревший `LoadedRooms` игнорируется» гоняет
  гард по `requestId` двумя сообщениями подряд, без асинхронности вообще.
- Путь retry-после-ошибки и баг переходов, который вскрыла живая проверка, оба
  **запинены здесь как регрессия**.

**`Scene` — тесты view (`src/scene.test.ts`).** Рендерит `view(model)` и проверяет по
ARIA-ролям / тексту (как Testing Library), привязано к петле `{ update, view }`, а не
к смонтированному компоненту:

```ts
Scene.scene({ update, view }, Scene.with(confirmed),
  Scene.expect(Scene.role('heading', { name: 'Booking confirmed' })).toExist(),
  Scene.expect(Scene.text('BK-20260622-COM')).toExist(),
)
```

**`validation.test.ts`** — обычный vitest над чистой `validate(model)`; возможен
только потому, что валидность — одна чистая функция, а не россыпь computed'ов и
гардов шаблона.

Контраст с Vue-версией: её тесты монтируют реальные компоненты на
`@vue/test-utils` + jsdom, **мокают API и await-ят** асинхронность, плюс Playwright
для e2e. Здесь моки и таймеры не нужны вообще — потому что эффект это значение,
которое можно проверить и «разрешить» вручную.

## Где закрыты дыры (карта файлов)

- Невозможные состояния убраны → `src/model.ts` (`DateSelection`, `RoomSelection`,
  `Availability`, `Phase`, `ValidBooking`)
- Единый источник валидности + проверка горизонта → `src/validation.ts`
- Staleness по request-id, исчерпывающие переходы, реальный submit → `src/update.ts`
- Эффекты как значения, команды → `src/command.ts`
- Доменные порты + граф зависимостей на Layer'ах (mock/http) → `src/services/` (`config.ts`, `httpClient.ts`, `roomCatalog.ts`, `availabilityCalendar.ts`, `reservations.ts`, `index.ts`)
- Типизированный канал ошибок (ADT + исчерпывающий перевод через `catchTags`) → `src/domain/errors.ts`, `src/services/`, `src/command.ts`
- Исчерпывающая обработка локали → `src/i18n.ts`
- Исчерпывающий роутинг фаза→шаг / фаза→view → `src/view/index.ts`
- Поведенческое доказательство → `src/validation.test.ts`, `src/story.test.ts`, `src/scene.test.ts`
